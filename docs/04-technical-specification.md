> **Version:** 0.1 | **Updated:** 2026-03-26
> **Repo:** `zip-captions-monorepo`

## 1. Code Style

### Language
English only — all code, comments, names, docs, commits. No exceptions. UI strings are externalized via ARB files.

### Linting
`very_good_analysis`. Zero warnings/errors. CI fails on violations.

```yaml
include: package:very_good_analysis/analysis_options.yaml
analyzer:
  exclude:
    - "**/*.g.dart"
    - "**/*.freezed.dart"
```

### Naming
- Files: `snake_case.dart`, one public class per file, filename matches primary export
- Classes: `PascalCase`
- Variables/functions/params: `camelCase`
- Constants: `camelCase` (Dart convention)
- Private: `_prefix`
- Riverpod providers: `camelCaseProvider` suffix (generated by `riverpod_generator`)

### File Organization (per package)
```
lib/
  src/           # All implementation — never import from src/ outside the package
    feature_x/
      feature_x.dart          # Barrel export
      feature_x_provider.dart
      feature_x_model.dart
      feature_x_widget.dart
  zip_core.dart  # Package barrel — re-exports public API from src/
```

### Import Rules
- Always use package imports (`package:zip_core/...`), never relative (`../`)
- No cross-package imports except through the public barrel file
- `zip_captions` and `zip_broadcast` depend on `zip_core` only — never on each other
- Sort imports: dart, flutter, packages, project (enforced by linter)

### Documentation
- All public APIs: dartdoc comment (`///`) with description
- No dartdoc on private members unless non-obvious
- No boilerplate comments ("This is a constructor", "Returns the value")
- **Inline comments are required** for non-obvious logic. Explain *why*, not *what*. Every conditional branch, algorithm step, or workaround should have a comment explaining its purpose.
- **No emojis anywhere in the codebase** — not in comments, commit messages, PR descriptions, docs, or string literals (except user-facing UI strings if explicitly required by a story)

## 2. Riverpod Patterns

### Code Generation
All providers use `riverpod_generator`. No hand-written `Provider`, `StateNotifierProvider`, etc.

```dart
// CORRECT — generated
@riverpod
class SttController extends _$SttController {
  @override
  Future<SttState> build() async { ... }
}

// WRONG — hand-written
final sttProvider = StateNotifierProvider<SttNotifier, SttState>((ref) => ...);
```

### Provider Types (when to use what)
- **Simple sync value:** `@riverpod` function (generates `Provider`)
- **Async data fetch:** `@riverpod` async function (generates `FutureProvider`)
- **Stream:** `@riverpod` Stream function (generates `StreamProvider`)
- **Mutable state with logic:** `@riverpod` class extending `_$ClassName` (generates `NotifierProvider` or `AsyncNotifierProvider`)
- **Parameterized:** use `.family` via function/constructor params — `riverpod_generator` handles this automatically

### AutoDispose
Default. All providers auto-dispose unless explicitly kept alive with `@Riverpod(keepAlive: true)`. Use `keepAlive` only for:
- App-lifetime services (auth state, settings, STT engine registry)
- Active STT sessions (disposed explicitly on stop)

### State Modeling
Use `freezed` for state classes with multiple states:

```dart
@freezed
class SttState with _$SttState {
  const factory SttState.idle() = _Idle;
  const factory SttState.listening({required String localeId}) = _Listening;
  const factory SttState.paused() = _Paused;
  const factory SttState.error(String message) = _Error;
}
```

### No Raw setState
No `setState()` or `ChangeNotifier` anywhere except inside leaf widgets with purely local UI state (e.g., animation controllers, text field focus). All business state goes through Riverpod.

## 3. Testing

### Philosophy
Strict. 80%+ code coverage per package. All public APIs tested. CI fails below threshold.

### Test Structure
```
test/
  src/
    feature_x/
      feature_x_provider_test.dart
      feature_x_model_test.dart
      feature_x_widget_test.dart
  helpers/          # Test utilities, mocks, fakes
    mocks.dart
    pump_app.dart   # Widget test helper with required providers
```

### Test Types
- **Unit tests:** All providers, models, business logic. Mock dependencies.
- **Widget tests:** All custom widgets. Use `pumpApp` helper that wraps with `ProviderScope` and required overrides.
- **Integration tests:** Critical flows (start captioning, save transcript, join broadcast). Run per-phase, not per-PR.
- **Golden tests:** Not required but encouraged for complex UI.

### Mocking
Use `mocktail` (not `mockito`). Create mocks in `test/helpers/mocks.dart`.

```dart
class MockSttEngine extends Mock implements SttEngine {}
```

### What Must Be Tested
- Every public method on every public class
- Every provider's build method and state transitions
- Error paths (not just happy path)
- Edge cases called out in the ADRs (e.g., device without auth can't save transcripts)

### What Doesn't Need Tests
- Generated code (`.g.dart`, `.freezed.dart`)
- Platform channel native code (tested via integration tests)
- Trivial getters/constructors with no logic

### Running Tests
```bash
melos run test          # All packages
melos run test --scope zip_core   # Single package
melos run coverage      # With coverage report
```

## 4. Platform Channels

### When to Use
Only when a Dart package doesn't exist or is inadequate for the requirement. Prefer existing packages. Document why a platform channel is needed.

### Structure
```
lib/src/feature_x/
  feature_x.dart                    # Public Dart API
  feature_x_platform_interface.dart # Abstract interface
  feature_x_method_channel.dart     # MethodChannel implementation

ios/Classes/
  FeatureXPlugin.swift

android/src/main/kotlin/.../
  FeatureXPlugin.kt

macos/Classes/
  FeatureXPlugin.swift

windows/
  feature_x_plugin.cpp

linux/
  feature_x_plugin.cc
```

### Rules
- Define an abstract platform interface in Dart — never call `MethodChannel` directly from business logic
- One channel per feature, named `com.zipcaptions.{feature}`
- All channel calls are `Future`-based (no synchronous channel calls)
- Handle `MissingPluginException` gracefully — the feature degrades, not crashes
- Platform code must not import or depend on any Dart business logic

## 5. Error Handling

### Error Flow
```
Platform code → PlatformException → Dart service → typed exception → Provider state → UI
```

### Rules
- Never catch generic `Exception` or `Object` — catch specific types
- Exception: top-level error boundaries in providers may catch `Exception` to set error state
- All errors that reach the user must have a human-readable message (localized)
- Never show stack traces to users
- Log errors with `dart:developer` `log()` — never `print()`
- Crash-level errors: report via crash reporting service (no transcript content, ever)

### Error Types
Define per-feature exception classes in `zip_core`:

```dart
sealed class SttException implements Exception {
  const SttException(this.message);
  final String message;
}

class SttEngineNotAvailable extends SttException { ... }
class SttPermissionDenied extends SttException { ... }
class SttInitializationFailed extends SttException { ... }
```

## 6. Dependency Management

### Approval Process
All new dependencies require documented justification. Before adding a package:

1. Check if the functionality can be achieved without a dependency
2. Check if an existing dependency already provides it
3. Evaluate: last updated, pub.dev score, GitHub issues, license, maintenance status
4. Add justification as a comment in the PR that introduces the dependency

### Version Pinning
- Use caret syntax (`^1.2.3`) for all dependencies
- Pin exact versions only if a specific version is known-broken
- Run `melos run upgrade` periodically, review changelogs before merging

### Approved Core Dependencies
These are pre-approved and do not need per-use justification:

| Package | Purpose |
|---------|---------|
| `flutter_riverpod` / `riverpod_generator` | State management |
| `freezed` / `freezed_annotation` | Immutable state classes |
| `json_serializable` / `json_annotation` | JSON serialization |
| `supabase_flutter` | Backend integration |
| `flutter_secure_storage` | Secure key storage |
| `flutter_webrtc` | WebRTC transport |
| `flutter_blue_plus` | BLE discovery |
| `speech_to_text` | Platform STT (if used) |
| `mocktail` | Test mocking |
| `very_good_analysis` | Linting |
| `build_runner` | Code generation |
| `melos` | Monorepo management |

### Prohibited Patterns
- No `dart:mirrors` (incompatible with AOT compilation)
- No `dart:html` in `zip_core` (must be platform-agnostic)
- No direct HTTP calls — use Supabase client SDK or a dedicated service class
- No global mutable state outside of Riverpod providers

### Code Review Requirements
- All PRs reviewed by at least one human

- Agent-generated code receives extra scrutiny on: dependency additions, platform channel implementations, encryption/auth code

## 8. Security Constraints

These are hard rules. Violations are blocking — no exceptions, no workarounds.

- **Never log, store, cache, or transmit transcript content** outside the user's device (except encrypted sync to Supabase Storage, encrypted by the client before transmission)
- **Never log, store, or inspect caption data in transit** through server infrastructure (Realtime relay, TURN)
- **Encryption keys never leave the device** except via explicit user-initiated key transfer (P2P, QR, hardware key)
- **No analytics, telemetry, or crash reports may contain transcript content** — not even partial, not even hashed
- **Server-side logs must explicitly exclude message payloads** — log metadata only (timestamps, connection events, error codes)
- **Devices without authentication cannot save transcripts** — enforce in code, not just policy
- **All Supabase tables use Row Level Security** — no table without RLS policies
- **OAuth provider list is not hardcoded** — architecture supports adding providers without code changes
- **Sensitive operations (key generation, encryption, decryption) use `dart:typed_data` and `package:cryptography` or equivalent audited library** — never roll custom crypto

## 9. Supabase Conventions

### Edge Functions
- TypeScript/Deno
- One function per file in `packages/zip_supabase/functions/`
- Named: `kebab-case`
- Must validate JWT on all authenticated endpoints
- Must return structured JSON errors, never raw exceptions

### Database
- Table names: `snake_case`, plural (`users`, `entitlements`, `broadcast_sessions`)
- Column names: `snake_case`
- All tables have `id` (UUID, primary key), `created_at`, `updated_at`
- RLS policies on every table — no exceptions
- Migrations in `packages/zip_supabase/migrations/`, numbered sequentially
- Migrations are forward-only — no editing existing migrations, create new ones to alter

### Realtime
- Channel naming: `broadcast:{session_id}`, `status:{broadcast_id}`, `signaling:{session_id}`
- Payload format: JSON with `type` field for message discrimination
- Caption payloads over Realtime must be E2E encrypted (client encrypts, server sees ciphertext)

## 10. Agent Development Rules

### Before Starting Work
1. Read `AGENTS.md` (root and package-level)
2. Read the linked story/issue
3. Read relevant ADRs referenced by the story
4. Read this document (you're reading it now)
5. Check `BACKLOG.md` or GitHub Projects board for task status

### Autonomy Boundaries
Agents may:
- Pick up stories with status "Ready"
- Write failing tests per story acceptance criteria
- Implement code to make tests pass
- Run lints, tests, and coverage checks
- Open PRs with completed checklist
- Create new files within the assigned package

Agents may NOT:
- Approve or merge PRs
- Modify files outside the assigned package without explicit story scope
- Add dependencies not on the approved list without human approval
- Modify security-critical code (encryption, auth, RLS policies) without human review of the approach BEFORE implementation
- Modify this document or any spec document
- Skip tests or reduce coverage
- Bypass linting rules with `// ignore` comments (except generated files)

### Task Scoping
- One story = one PR
- If a story is too large for one PR, split it into sub-stories before starting
- If blocked on another story, stop and document the blocker in the issue — do not work around it

### Code Generation
After modifying any file that uses `freezed`, `riverpod_generator`, or `json_serializable`, run:
```bash
melos run generate  # or: dart run build_runner build --delete-conflicting-outputs
```
Commit generated files alongside source changes.

## 11. Package Responsibilities

### `zip_core`
Shared library. No Flutter UI code (no `Widget`, no `BuildContext` except in optional widget utilities). Contains:
- STT engine interface and implementations
- Translation engine interface and implementations
- Audio capture abstraction
- Caption bus (pub-sub stream)
- Encryption/decryption services
- Storage abstraction (local DB, secure storage)
- Models and data classes
- Settings/configuration
- BLE discovery protocol
- Transport layer (WebRTC, local Wi-Fi, BLE GATT)
- Auth wrapper
- Localization (ARB files, shared strings)
- Entitlement checking

### `zip_captions`
Personal user app (Alex + Sam personas). Depends on `zip_core`. Contains:
- Mobile and desktop UI (screens, widgets, navigation)
- App-specific Riverpod providers
- Conversation translation mode
- Caption overlay (premium, desktop)
- Viewer broadcast join flow
- Platform-specific configuration (iOS, Android, macOS, Windows, Linux, web)

### `zip_broadcast`
Broadcaster app (Jordan persona). Depends on `zip_core`. Contains:
- Desktop UI (screens, widgets, navigation)
- Broadcast session management UI
- OBS WebSocket integration
- Browser source server
- Multi-output configuration
- Bilingual display mode
- Operator mode
- Platform-specific configuration (macOS, Windows, Linux, web)

### `zip_supabase`
Backend. Contains:
- Edge Functions (TypeScript/Deno)
- Database migrations (SQL)
- Seed data
- RLS policies
